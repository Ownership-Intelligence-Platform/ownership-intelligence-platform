Starting (2/6) _Decide on crawler necessity_

我先把“客户尽调”的业务目标梳理清楚，并给出端到端的落地方案与优先级，重点回答“要不要做爬虫”和“怎么做自动化”。

## 总体结论：是否需要做爬虫

- 首选不是爬虫。优先级应为：
  1. 官方接口/开放数据/公开订阅源；
  2. 合规的数据服务商（商业数据 API/批量文件）；
  3. 开放网页的合规抓取（尊重 robots.txt/站点条款、节流、仅采集必需的公共信息），将其定位为“补齐缺口”的手段。
- 原因：合规与稳定性是 CDD 的生命线。大量站点（如司法文书网、招聘平台）对自动化抓取限制严格、存在验证码/风控、条款限制。用“可授权、可付费、可审计”的数据源，能极大提高可用性与合规确定性。
- 因此，策略是“API/数据服务为主，爬虫为辅”，并将爬虫纳入数据治理（采集范围、速率、留痕、删除、更新周期、法律审查）。

## 目标业务架构（不谈代码，谈流程和能力分层）

- 数据采集层（Source Connectors）
  - 官方/开放：国家及省级企业信用信息公示、政府采购网、部分监管公告 RSS、联合国/欧盟/OFAC 制裁清单等；
  - 商业供应商：企业工商司法（如天/启/企等合规供应商）、PEP/制裁（Refinitiv/Dow Jones/Factiva 等）、招投标聚合、招聘聚合；
  - 合规抓取：仅用于无 API 的公开页，如部分省级采购公告、部分招聘列表页，设置频控与缓存。
- 实体解析层（Entity Resolution）
  - 名称归一化：中文/英文/拼音/简称/同义前后缀（集团/有限公司/有限责任公司等）；
  - 强特征对齐：统一社会信用代码/工商注册号/官网域名/电话/地址；
  - 歧义消解：同名企业的省份/行业/法定代表人/历史名称佐证，给出“候选列表+置信度”，保留人工确认环节。
- 标准化与图谱层（Normalization & Knowledge Graph）
  - 统一数据模型：企业/个人/政府机构/招采项目/判决文书/处罚决定/招聘岗位等实体，及关系边：中标、被诉、处罚、雇佣、股权、任职、分支、供应/客户等；
  - 时间、来源、证据链留痕：每个事实保留“采集时间、来源 URL/文件、摘要/截图签名”。
- 风险识别与打分（Rulebook + Analytics）
  - 规则库（可配置）：司法纠纷频率/败诉、失信被执行、行政处罚、经营异常、招采异常（废标/黑名单）、招聘信号（涉敏高频岗位、业务关键词）、负面舆情、制裁/PEP 命中、与高风险实体的间接关联等；
  - 权重与阈值：按场景（开户准入/强化尽调/AML）设不同阈值与解释模板；
  - 图分析：担保圈、股权穿透（UBO）、层级深度相关性、间接风险传导。
- 人工在环（Human-in-the-loop）
  - 两个关键节点必须人工：实体对齐确认（避免同名误判）；高风险结论审批；
  - 一键纠错/忽略来源/标注误报，反哺规则与匹配词典。
- 报告与闭环（CDD Snapshot & Case Workflow）
  - 统一报告模板：概览、股权穿透图、司法/处罚/招采/招聘/舆情证据清单、关键结论与建议（通过/人工复核/拒绝）；
  - 工单化：新客户触发采集任务、进度可视、结论落库、留痕可审计、定期增量刷新。

## 数据源清单与接入优先级（中国场景为例）

优先接 API 或商用数据，再以开放源补充，最后才考虑合规抓取。

- 工商/登记/变更
  - 国家企业信用信息公示系统（含省局跳转）、全国组织机构、信用中国部分公示；
  - 商业供应商：企业主体信息、历史名称、股权/高管/分支/变更、关联风险事件。
- 司法/执行/失信
  - 中国裁判文书、执行信息公开网、失信被执行人名单（多存在访问限制，建议走供应商镜像/数据产品）；
  - 商业供应商：判决标签化、案由归类、角色识别（被告/原告/第三人）。
- 行政处罚/经营异常
  - 市监总局、省市市场监管公告、信用中国行政处罚/异常公示；
  - 行业监管（银保监/证监/网信办等）公开处罚通报。
- 招投标/政府采购
  - 中国政府采购网、各省市交易中心（格式分散）；
  - 商业聚合：历史中标、废/流标、黑名单、供应商画像。
- 招聘平台信号
  - 智联、前程无忧、BOSS 直聘等（商业数据或合作接口优先；如无则仅抓取公开且允许页面的摘要信息）；
  - 信号项：岗位类型（风控/合规/催收/博彩关键词等）、地域扩张、薪资区间异常、业务关键词。
- 舆情/新闻/公告
  - 官媒/主流媒体 RSS/开放 API、企业官网公告、监管公告；
  - 商业舆情服务：更好的去重/情感倾向/主题聚类。
- 制裁/PEP
  - UN/OFAC/EU 等公开清单；PEP 数据需商业化数据服务（World-Check、Refinitiv、Dow Jones 等），注意许可证。
- 其他合规公开数据
  - 海关/税务部分公示、行业黑名单、认证/许可目录等。

## 实体解析与 Name Screening（升级点）

- 多模态匹配策略：
  - 字面匹配：全称/历史名/英文名/别名/缩写；
  - 结构化强特征：统一社会信用代码/注册号/成立日期/地址/法人；
  - 跨语匹配：中文-拼音-英文别名映射、序列调换（Zhang San/San Zhang）、同音异字词典；
  - 上下文限制：同省/同市、相同行业/同法人，显著提升精度；
  - 得分+置信区间：高分自动通过，中分进入人工确认，低分丢弃或需要更多特征。
- 维护词典与知识：
  - 组织形态后缀、地域别名、常见简称库；
  - 风险词表（用于招聘、招采文本、新闻标题的模式识别）。

## 风险规则与评分（示例）

- 极高风险（直接拒绝/升级尽调）
  - 制裁命中、PEP 本人/直系关联、失信被执行（新近）、严重行政处罚（涉金融/反洗钱/数据安全）、经营异常/吊销。
- 高风险（人工复核）
  - 近 12 个月多次被诉/败诉、被纳入限制高消费、重大招标废标/黑名单、频繁招聘敏感岗位（催收/博彩/现金贷等）且与主营不一致、负面舆情集中爆发。
- 中风险（关注）
  - 异常人事/股权频繁变更、集中异地扩张、关联方风险传导（两跳内高风险节点）。
- 打分与解释
  - 每条规则给权重、时效衰减、证据链引用（来源/时间/链接/摘要），出具可审计解释。

## 新客户进入的端到端流程（业务视角）

1. 立案与目标确认：输入客户名称/统一社会信用代码/地区/官网域名等；
2. 实体对齐：生成候选实体与置信度，人工一键确认；
3. 多源检索：按优先级并发查询（API/供应商/开放源），对“抓取型”源进行节流与缓存；
4. 标准化入库：结构化实体与关系，保留原文与证据链接；
5. 风险分析：规则打分+图关联+舆情聚类，形成结论与建议动作；
6. 人工复核：高风险或歧义样本进入复核队列，支持驳回/豁免/补充证据；
7. 出具报告：统一模板（中英可选）、图谱快照、证据清单、审批留痕；
8. 持续刷新：重要客户设定监控策略（每日/每周增量），异常触发预警。

## 合规与数据治理（关键红线）

- 法律与条款：遵守《数据安全法》《个保法》、站点使用条款/robots 协议；对受限网站避免非授权采集；
- 最小化与可审计：只采集业务必需，记录来源/时间/处理过程；提供删除与更正机制；
- 速率与缓存：统一出站网关，限流与重试策略，缓存 TTL 与再核验周期；
- 供应商管理：合同许可范围、用途限制、再分发限制、数据质量 SLA；
- 安全：敏感数据加密存储，访问最小权限，操作留痕，定期审计。

## 分阶段落地建议（12 周样例）

- 0-2 周：需求细化与合规评审
  - 明确名单源与许可、确定首批商业数据供应商、冻结指标与报表模板。
- 3-6 周：MVP（可出报告）
  - 打通：企业登记信息（供应商/官方）、司法与处罚（供应商）、政府采购（国家级）、新闻/公告；
  - 上线：实体对齐+规则打分+模板化报告；人工复核闭环。
- 7-10 周：覆盖扩展与稳定性
  - 招聘信号源（至少 1 家）、省级采购/交易中心增补、黑名单/PEP/制裁接入；
  - 增量刷新、预警、误报处理与词典维护流程。
- 11-12 周：优化与评估
  - 评估准确率/通过率/人均处理时长/成本；调整规则与权重，规划下一阶段（UBO/离岸、更多图算法）。

## KPI 与验收标准

- 周期：单客户自动化尽调 ≤30 分钟（不含人工复核）；
- 覆盖：首批优先源覆盖率 ≥80%（相对人工“常查清单”）；
- 准确：实体对齐 Precision≥95%、Recall≥90%（在抽样标注集上）；
- 成本：人均复核工作量下降 ≥50%，报告合规通过率 ↑；
- 可审计：每条结论可回溯到原始证据。

## 团队与成本（概览）

- 角色：数据治理/合规、数据源集成、实体解析与 NLP、风险策略、产品与流程、运维与安全；
- 成本：数据供应商与许可是主要支出；自行抓取不是“免费”，需要运维、风控和合规投入。

- 固定首版报告模板与规则权重，准备一批历史案例做离线评估。
